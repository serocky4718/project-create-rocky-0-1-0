# Failure Recovery Rules

本文件定义常见失败场景的处理方式。

核心原则：失败时先停下来说明状态，不静默重试高风险操作，不用破坏性命令掩盖问题。

## 通用失败处理

遇到失败时，先输出：

```text
失败发生在哪一步
错误信息或可观察现象
可能原因
已经尝试过什么
不会继续做什么高风险操作
建议下一步
需要用户确认的问题
```

## GitHub 建库失败

常见原因：

- 网络问题。
- GitHub 未登录或 token 失效。
- 权限不足。
- 仓库名冲突。
- 当前 GitHub 工具不可用。

处理规则：

- 不重复创建多个仓库。
- 不随意切换账号或组织。
- 如果只是仓库名冲突，按 `github-automation-rules.md` 追加后缀。
- 如果是权限或登录问题，提示用户完成 GitHub 登录或授权。
- 在 `HANDOFF.md` 记录失败状态和下一步。

## 写入文件失败

常见原因：

- 目录不存在。
- 文件被占用。
- 权限不足。
- 路径写错。

处理规则：

- 先确认目标目录。
- 目录不存在时，可以创建目录，但要说明会创建什么。
- 不覆盖已有文件，除非用户确认。
- 写入失败后，不假装文件已生成。

## 验证命令失败

常见原因：

- 依赖未安装。
- 命令不存在。
- 环境变量缺失。
- 测试本身失败。
- 构建配置缺失。

处理规则：

- 区分“环境问题”和“代码问题”。
- 不默认安装新依赖，除非用户确认或项目规则允许。
- 记录完整命令、失败摘要、建议修复路径。
- 如果是环境缺失，交给 Builder 或 DevOps/GitHub 判断。

## 部署失败

常见原因：

- 环境变量缺失。
- 平台未登录。
- 构建失败。
- 权限不足。
- 费用或额度限制。

处理规则：

- 不重复部署刷日志。
- 不修改生产配置绕过错误。
- 先说明失败原因和风险，再让用户确认下一步。

## 成功标志

```text
失败被明确分类
没有执行破坏性补救
用户知道下一步要做什么
HANDOFF.md 记录当前失败状态
```
